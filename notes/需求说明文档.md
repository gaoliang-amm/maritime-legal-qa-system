# 船舶海事法律智能问答系统 — 需求说明文档

## 1. 项目概述

### 1.1 项目名称

船舶海事法律智能问答系统

### 1.2 项目背景

船员权益保障领域法律条文复杂，包括《中华人民共和国海商法》（2025年修订）和《中华人民共和国海上交通安全法》（2021年修订）等核心法律。从业人员需要快速检索和理解相关法律条文，但传统查阅方式效率低下。

### 1.3 项目目标

构建一个智能法律问答系统，能够：

- 解析法律文档并建立向量索引
- 准确理解用户关于船员权益的法律问题
- 从知识库中检索相关法律条文
- 生成专业、有依据的法律解答
- 支持多轮对话和上下文理解

### 1.4 目标用户

- 船员、海员
- 航运公司法务部门
- 海事律师、法律顾问
- 海事法院工作人员
- 船员服务机构

## 3. 技术架构

### 3.1 总体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                          浏览器                                  │
└────────────────────────────┬────────────────────────────────────┘
                             │ HTTP 请求
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  Web 层：FastAPI + Gunicorn + UvicornWorker                     │
│  (请求校验、Session 管理、异常处理、上下文注入)                  │
└────────────────────────────┬────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  调度层：LangGraph Workflow Engine                               │
│  (状态初始化、分支决策、子图协调、结果汇聚)                      │
└────────────────────────────┬────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  执行层：Task 状态机                                             │
│  (任务生命周期管理、错误处理与重试、回调通知)                    │
└────────────────────────────┬────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  输出层：流式渲染                                                │
│  (Markdown 渐进渲染、引用高亮、SSE 连接管理)                    │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 技术栈

| 组件 | 选型 | 用途 |
|------|------|------|
| Web 框架 | FastAPI + Gunicorn + UvicornWorker | HTTP 服务、请求校验 |
| 工作流引擎 | LangGraph | 状态初始化、分支决策、子图协调 |
| 任务管理 | Task 状态机 | 任务生命周期管理、错误处理与重试 |
| 输出渲染 | Markdown + SSE | 流式渲染、引用高亮 |
| 向量模型 | BGE-M3 | dense + sparse 混合向量 |
| 重排序 | BGE-reranker-large | 精细化排序 |
| 向量库 | Milvus | 混合检索 |
| 文档数据库 | MongoDB | 历史对话 |
| PDF解析 | MinerU | PDF→Markdown |

## 4. 流程设计

### 4.1 数据入库流程

```
用户上传文档
     │
     ▼
 ┌───────────┐
 │ FastAPI   │  接收上传请求
 └─────┬─────┘
       │
       ▼
 ┌───────────┐
 │ 文档解析   │  上传 PDF，调用 MinerU 解析为 Markdown
 └─────┬─────┘
       │
       ▼
 ┌───────────┐
 │ 文本切分   │  按条款级别切分文本
 └─────┬─────┘
       │
       ▼
 ┌───────────┐
 │ 建立索引   │  BGE-M3 生成向量，存入 Milvus
 └───────────┘
```

**详细步骤**：

1. **用户上传文档**：通过浏览器上传 PDF 文件
2. **FastAPI 接收**：校验文件格式，返回任务 ID
3. **文档解析**：调用 MinerU 将 PDF 解析为 Markdown 格式
4. **文本切分**：按条款级别切分，保留章节结构信息
5. **向量化**：调用 BGE-M3 模型生成 dense + sparse 混合向量
6. **建立索引**：将向量和元数据存入 Milvus

### 4.2 数据检索流程

```
用户输入问题
     │
     ▼
 ┌───────────┐
 │ rewriter  │  问题改写
 └─────┬─────┘
       │
       ▼
 ┌───────────┐
 │ 问题路由   │  意图识别 + 多轮改写 + 融合判断
 └─────┬─────┘
       │
       ▼
 ┌───────────┐
 │ 问题分类   │  确定问题类型
 └─────┬─────┘
       │
       ▼
 ┌───────────┐
 │ 查询改写   │  优化查询表达
 └─────┬─────┘
       │
       ▼
 ┌───────────┐
 │ 短文本查询  │  向量检索
 └─────┬─────┘
       │
       ▼
 ┌───────────┐
 │ 结果重排   │  BGE-reranker 精排
 └─────┬─────┘
       │
       ▼
 ┌───────────┐
 │ 引用溯源   │  标注来源条款
 └─────┬─────┘
       │
       ▼
 ┌───────────┐
 │ 结果输出   │  生成最终答案
 └───────────┘
```

**详细步骤**：

1. **rewriter**：接收用户问题，执行初步改写
2. **问题路由**：意图识别、多轮改写、融合判断
3. **问题分类**：确定问题类型（法律查询/解释咨询/对比分析/闲聊）
4. **查询改写**：结合上下文优化查询表达
5. **短文本查询**：BGE-M3 生成向量，Milvus 混合检索
6. **结果重排**：BGE-reranker-large 精细化排序
7. **引用溯源**：标注答案依据的法律条款
8. **结果输出**：生成最终答案

## 5. 模块设计

### 5.1 模块总览

```
                    ┌─────────────┐
                    │   用户提问   │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │  问题路由    │ │  知识检索    │ │  回答生成    │
    │ (意图识别、 │ │ (召回、重排) │ │ (重排、提示词│
    │  多轮改写、 │ │              │ │  构建、引用  │
    │  融合判断)  │ │              │ │  溯源、生成) │
    └─────────────┘ └─────────────┘ └─────────────┘
```

### 5.2 问题路由模块

**功能**：接收用户问题，执行意图识别、多轮改写和融合判断

**核心组件**：

#### 5.2.1 意图识别节点

- **输入**：用户问题、对话历史
- **处理**：LLM 分析问题意图
- **输出**：意图分类

```python
class IntentType(Enum):
    LEGAL_QUERY = "legal_query"        # 法律条文查询
    EXPLANATION = "explanation"        # 法律解释咨询
    COMPARISON = "comparison"          # 对比分析
    CHITCHAT = "chitchat"             # 闲聊
```

#### 5.2.2 多轮改写节点

- **输入**：当前问题、对话历史
- **处理**：结合上下文改写问题，解决指代、省略
- **输出**：改写后的完整问题

#### 5.2.3 融合判断节点

- **输入**：原始问题、改写问题、意图
- **处理**：决定是否合并历史上下文
- **输出**：最终查询语句

### 5.3 知识检索模块

**功能**：从向量数据库中召回相关法律条文

**核心组件**：

#### 5.3.1 向量召回节点

- **输入**：查询语句
- **处理**：BGE-M3 生成向量，Milvus 混合检索（dense + sparse）
- **输出**：候选文档列表（Top-K）

```python
RETRIEVAL_CONFIG = {
    "top_k": 20,
    "vector_type": "hybrid",      # dense + sparse
    "similarity_threshold": 0.5
}
```

#### 5.3.2 结果重排节点

- **输入**：候选文档列表、查询语句
- **处理**：BGE-reranker-large 精排
- **输出**：排序后的文档列表（Top-N）

```python
RERANK_CONFIG = {
    "model": "BAAI/bge-reranker-large",
    "top_n": 5,
    "max_length": 512
}
```

### 5.4 回答生成模块

**功能**：生成带引用的法律解答

**核心组件**：

#### 5.4.1 文档重排节点

- **输入**：检索结果
- **处理**：按相关性二次排序
- **输出**：精排后的文档

#### 5.4.2 提示词构建节点

- **输入**：用户问题、检索结果
- **处理**：组装提示词（含法律条文上下文）
- **输出**：完整提示词

#### 5.4.3 引用溯源节点

- **输入**：生成的答案、源文档
- **处理**：标注答案依据的条款
- **输出**：带引用标记的答案

#### 5.4.4 短文本生成节点

- **输入**：提示词
- **处理**：LLM 生成答案
- **输出**：法律解答

## 6. 提示词设计

### 6.1 写作者提示词

用于生成流畅、专业的法律解答：

```
你是一位专业的海事法律顾问，擅长解读船员权益保障相关法律。

写作要求：
1. 使用准确的法律术语
2. 语言简洁明了，便于船员理解
3. 必须引用具体法律条款
4. 如有多个相关条款，按重要性排序
```

### 6.2 分析框架提示词

用于结构化分析法律问题：

```
请按以下框架分析用户的法律问题：

1. 【问题定性】：识别问题涉及的法律关系
2. 【适用法条】：列出相关法律条款
3. 【条款解读】：解释条款含义
4. 【结论建议】：给出明确结论和建议
```

### 6.3 角色扮演提示词

用于模拟不同场景的问答：

```
你现在扮演以下角色回答问题：
- 角色：海事律师
- 专长：船员劳动争议
- 语言风格：专业、严谨、有理有据

回答时请注意：
- 先给出结论，再展开分析
- 引用具体法条作为依据
- 如有争议，说明不同观点
```

```http
POST /api/upload
Content-Type: multipart/form-data

- file: PDF文件

Response:
{
    "task_id": "uuid",
    "status": "processing",
    "message": "文档处理中..."
```

### 7.2 问答接口

```http

data: {"answer": "...", "references": [...]}
```

## 8. 非功能需求

### 8.1 性能需求

| 指标 | 目标值 |
|------|-------|
| 首次响应时间 | < 2秒 |
| 完整答案生成 | < 10秒 |
| 向量检索延迟 | < 100ms |

### 8.2 质量需求

| 指标 | 目标值 |
|------|-------|
| 条款检索准确率 | ≥ 90% |
| 答案相关性 | ≥ 85% |
| 来源标注准确率 | ≥ 95% |

### 8.3 向量相似度阈值

```python
SIMILARITY_THRESHOLD = 0.5  # 低于此阈值的结果不返回
```

## 9. 优化机制

### 9.1 用户反馈机制

- 用户对回答进行评价（有用/无用）
- 记录反馈数据用于后续优化
- 定期分析低分回答，优化提示词和检索策略

### 9.2 向量相似度阈值

- 设置相似度阈值（0.5），低于阈值的结果不返回
- 避免返回不相关的法律条文
